Doing computer proof intuitively

This page is largely a summary of a talk I gave at Churchill College Cambridge in October 2020 entitled "A graph of all mathematics".

The Curry-Howard correspondence

The Curry-Howard corresponence is a very cool and important result in theoretical computer science. In short it tells us that mathematical proofs are basically the same thing as functional programs, which allows programmers to steal all of pure mathematics' best ideas.

I'm going to run through a simple example because it can be quite confusing to understand at first. Let's say we know (1) that if it is raining it will be muddy, and (2) if it will be muddy we should wear boots. We want to prove that (3) if it is raining we should wear boots.

We can represent (1) and (2) as functions. f1 takes knowledge that it it is raining and returns knowledge that it will be muddy. f2 takes knkowledge that it will be muddy and returns knowledge that we should wear boots.

Now to write a function that takes knowledge that it is raining and returns knowledge that we should wear boots is trivial. def f3(x): f2(f1(x)).

Too much Curry-Howard!

The Curry-Howard corrresponence is the underlying basis for most of the computer proof assistants in use today. But the Curry-Howard correspondence only takes something most people understand (maths) and turn it into something that almost no-one understands (functional programming). As a result, only funcional programmers are getting excited about computer proof at the moment!

So I'm going to present what I think is a more natural approach. It's based on graphs (the dots-and-lines kind) but the graphs are laid out very similar to how we lay proofs out on paper.
